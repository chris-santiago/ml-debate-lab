# Self-Debate Experiment v5

v5 is a redesign of the debate protocol evaluation experiment, motivated by calibration failures in v2–v4 where benchmark cases were too easy and produced near-ceiling results. The core fix is a dedicated synthetic case generation pipeline that produces harder, more controlled benchmark cases before the main experiment runs.

**Current status:** Case generation phase — generating and calibrating the benchmark case library. Main experiment phases (debate execution, scoring, analysis) have not started.

## INVESTIGATION_LOG.jsonl

A sequential, append-only audit trail. Every significant action in the v5 experiment — writing an artifact, running a stage, fixing a bug, making a design decision — gets a JSONL entry.

### Schema

```json
{
  "ts":         "2026-04-09T03:05:35Z",   // ISO 8601 timestamp (UTC)
  "step":       "v5-pipeline",             // Phase or component identifier
  "seq":        3,                          // Monotonically increasing; enforced by log_entry.py
  "cat":        "write",                   // Category — see valid values below
  "action":     "add_artifact",            // Finer-grained label within category
  "detail":     "Added CASE_GENERATION_REPORT.md ...",  // Human-readable description
  "artifact":   "self_debate_experiment_v5/CASE_GENERATION_REPORT.md",  // File(s) affected
  "duration_s": null,                      // Optional: elapsed seconds for timed operations
  "meta":       {}                         // Optional: structured metadata
}
```

**Valid `cat` values:** `audit`, `debate`, `decision`, `exec`, `gate`, `read`, `review`, `subagent`, `workflow`, `write`

### Writing entries

Always use `log_entry.py` — never append manually. Manual appends break sequence monotonicity and schema compliance.

```bash
cd self_debate_experiment_v5

uv run plan/scripts/log_entry.py \
  --step "v5-pipeline" \
  --cat "write" \
  --action "add_artifact" \
  --detail "Description of what was done and why." \
  --artifact "relative/path/to/file.py"
```

The script enforces `seq` ordering by reading the current max before writing. Running two concurrent writes is unsafe.

## Benchmark Case Files

Each `cases_NNN-MMM.json` is a JSON array of benchmark cases, one entry per case. Cases are generated by the pipeline in `synthetic-candidates/pipeline/` — see `pipeline/README.md` for generation commands.

### Case schema (abbreviated)

```json
{
  "case_id":          "eval_scenario_100",
  "hypothesis":       "Concrete claim being tested",
  "domain":           "Healthcare / clinical ML",
  "ml_task_type":     "Binary classification",
  "task_prompt":      "The full experiment design proposal — what debate agents see",
  "correct_verdict":  "critique | defense_wins",
  "num_corruptions":  0 | 1 | 2 | "many",
  "planted_issues":   [...],              // One entry per planted flaw
  "must_find_issue_ids": ["issue_001"],  // Issues debate must surface
  "must_not_claim":   [...],             // Legitimate choices that must not be criticized
  "acceptable_resolutions": [...],       // What a correct debate outcome looks like
  "sound_design_reference": "...",       // Correct design narrative (not shown to agents)
  "_pipeline": {
    "mechanism_id":   "mech_001",
    "num_corruptions": 1,
    "proxy_mean":     0.42,              // Smoke test difficulty signal; lower = harder
    "smoke_scores":   {"IDR": 0.0, "IDP": 1.0, "FVC": 0.0}
  }
}
```

**`proxy_mean` interpretation:** Mean of IDR, IDP, FVC from the blind Sonnet smoke test. Lower values mean Sonnet got the verdict wrong or missed planted flaws — the case is harder. Use `--max-proxy 0.83` in `select_cases.py` to filter out easy cases.

### Selected case files

`selected_cases_*.json` are outputs of `pipeline/select_cases.py` — balanced subsets stratified across `(correct_verdict × corruption_tier × domain × ml_task_type)` with a difficulty filter applied. These are the intended inputs to the main experiment phases.

## Diagnostics

### `ISSUE_TRACKER.md`

The canonical re-entry point for calibration work. Lists all open and resolved issues with root causes, fixes applied, and empirical validation results. Check this first when resuming work on the pipeline.

### `CALIBRATION_DIAGNOSTIC.md`

Deep-dive root cause analysis of why v1–v4 smoke tests showed near-ceiling scores (flaws too detectable). Covers the full diagnostic process, design options considered, and rationale for choices made. Written 2026-04-07.

### `CALIBRATION_CHANGES_SUMMARY.md`

Implementation checklist tracking all pre-Phase-0 fixes applied across calibration rounds — prompt changes (Levers A and B), rubric adjustments, pipeline changes. Cross-references ISSUE_TRACKER entries.

### `CASE_FORMAT_DIAGNOSTIC.md`

Documents a critical compliance failure discovered during smoke test V4: the LLM generator was not populating `task_prompt` correctly. This made cases structurally unevaluable and invalidated several early calibration rounds.

### `V5_PLAN_REVISION_SUMMARY.md`

Summarizes all revisions made to `plan/` (phases and scripts) during the calibration period — what changed, what was added, and why.

### Smoke test snapshots

The `HAIKU_SMOKE_TEST_*.md` and `REAL_PAPER_CASES_SMOKE_TEST*.md` files are result snapshots from successive calibration rounds, preserved for comparison. They show the progression from early real-paper-sourced cases through the ARCH-1 pipeline batches.

## Plan

### `plan/PLAN.md`

The full 11-phase execution plan for the main experiment (debate runs, scoring, analysis, reporting). Written before the ARCH-1 pipeline was built — references some old case file paths that have since been superseded by the `synthetic-candidates/` pipeline output.

**Phases overview:**

| Phase | Purpose |
|-------|---------|
| 0 | Setup — merge cases, copy scripts, validate environment |
| 1 | Case verification — structural checks before agents see any case |
| 2 | Hypothesis documentation, benchmark prompt generation |
| 3 | Pre-registration and rubric lock |
| 4 | Protocol self-review |
| 5 | Scoring engine build (`self_debate_poc.py`, 5 conditions) |
| 5.5 | Difficulty gate — 15-case pilot; verify ≥6/10 hard cases score < 0.55 |
| 6 | Main benchmark run — 5 conditions × all cases |
| 6.5 | External benchmark evaluation |
| 6.75 | Difficulty validation probe |
| 7 | Scoring and statistics (bootstrap CIs, Wilcoxon, sensitivity) |
| 8 | Analytical artifacts (conclusions, sensitivity, ensemble analysis) |
| 8.5 | Pre-report coherence audit |
| 9 | Cross-vendor scorer |
| 9.5 | Post-run audit agent |
| 10 | Final report, peer review, synthesis |
| 10.5 | Post-report coherence audit |
| 10.75 | Technical report (publication-ready) |
| 11 | Final verification and commit |

### `plan/phases/`

Per-phase execution instructions — one file per phase. These are the step-by-step prompts that drive Claude Code session execution for each phase.

### `plan/scripts/`

Supporting Python scripts for the main experiment. All use PEP 723 inline headers; run with `uv run`.

| Script | Purpose |
|--------|---------|
| `log_entry.py` | Append entries to `INVESTIGATION_LOG.jsonl` — always use this, never manual append |
| `self_debate_poc.py` | Main scoring engine — 5 debate conditions, IDR/IDP/FVC/DC/ETD metrics |
| `validate_cases.py` | Structural validation of case JSON before experiment runs |
| `validate_raw_schema.py` | Schema check on raw pipeline output before assembly |
| `filter_verified_cases.py` | Filter to only verified/accepted cases |
| `stats_analysis.py` | Bootstrap CIs, Wilcoxon signed-rank tests |
| `sensitivity_analysis.py` | Fair-comparison lift decomposition |
| `difficulty_validation.py` | Spearman rho between proxy_mean and experiment scores |
| `within_case_variance.py` | Variance analysis across runs within each case |
| `coherence_audit.py` | Pre-report cross-document consistency check |
| `post_report_coherence_audit.py` | Post-report cross-document consistency check |
| `cross_model_scorer.py` | Cross-vendor scoring (Phase 9) |
| `write_preregistration.py` | Generate PREREGISTRATION.json (locked before Phase 6) |
| `check_isolation.py` | Verify debate conditions are properly isolated |
