# Self-Debate Protocol — Experiment v3

## Hypothesis

This experiment tests whether an isolated self-debate protocol — where ml-critic and ml-defender each receive only the task prompt independently, with an orchestrator synthesizing both outputs — achieves a benchmark aggregate score at least +0.10 higher than a single-pass baseline on 49 synthetic ML reasoning tasks with known ground-truth verdicts. The mechanism under test is asymmetric role-instruction ensemble: the Critic is biased toward flaw detection while the Defender is biased toward exoneration, and the synthesized combination is expected to outperform either agent alone. A multi-round adversarial exchange condition (where the Defender sees the Critique before responding) and an ETD-constrained ensemble condition (compute-matched, no Defender) are run in parallel to isolate the structural contribution of role-assignment versus adversarial exchange versus compute budget.

## Quickstart

```bash
uv run self_debate_poc.py
```

Requires: agent outputs in `v3_raw_outputs/` (588 files: 49 cases × 4 conditions × 3 runs). Run Phase 6 first to generate raw outputs.

Optional flags:
```bash
uv run self_debate_poc.py --cases benchmark_cases_verified.json --output v3_results.json
```

## Pipeline

```
benchmark_cases_verified.json
        |
        v
  [Phase 6 — Agent Dispatch]
  Four conditions, 3 runs each:
    isolated_debate  — Critic + Defender (isolated), synthesized
    multiround       — Critic -> Defender (sees critique) -> 4-round exchange
    ensemble         — 3x Critic (no Defender), ETD-constrained synthesizer
    baseline         — 1x Critic, no synthesis
        |
        v
  v3_raw_outputs/
  {case_id}_{condition}_run{n}.json  (588 files)
        |
        v
  [Phase 7 — Scoring]
  uv run self_debate_poc.py
    score_run() per case per condition per run
    aggregate_runs() — mean, std, pass count across 3 runs
    convergence metric — critic_raw_verdict vs. defender_raw_verdict
        |
        v
  v3_results.json          — aggregate + per-case scores
  evaluation_results.json  — per-case pass/fail, issues, attribution
        |
        v
  [Phase 8 — Statistics and Analysis]
  uv run stats_analysis.py          — bootstrap CI, filtered DRQ comparison
  uv run sensitivity_analysis.py    — honest lift, corrected means
  uv run difficulty_validation.py   — score by difficulty level
        |
        v
  stats_results.json
  sensitivity_analysis_results.json
  difficulty_validation_results.json
  within_case_variance_results.json
  *.png (score distributions, lift by category, ETD comparison)
        |
        v
  [Phase 9 — Reporting]
  CONCLUSIONS.md  — hypothesis verdicts, convergence diagnostic
  REPORT.md       — full results
  TECHNICAL_REPORT.md — publication-ready summary
```

## Expected Outputs

After a complete run:

- `v3_results.json` — primary results: benchmark means by condition, lift, pass rates, per-case breakdown
- `evaluation_results.json` — per-case pass/fail, found/missed must_find issues, false positive issues, failure attribution
- `CONCLUSIONS.md` — primary hypothesis verdict (H1), secondary hypothesis verdicts (H2, H3, H4), convergence diagnostic for isolated_debate
- `SENSITIVITY_ANALYSIS.md` — honest lift (corrected for DC=0.0 and DRQ cap), rubric variant analysis
- `ENSEMBLE_ANALYSIS.md` — ETD comparison on mixed-position cases, defense_wins DC fraction
- `TECHNICAL_REPORT.md` — publication-ready summary with all pre-registered criteria and outcomes

## Scope Exclusions

The following are explicitly out of scope for this experiment:

- **Case generation:** All benchmark cases were generated by an external model (GPT-5.4 on Perplexity Computer). Claude Code validates and executes only.
- **Rubric modification after execution begins:** evaluation_rubric.json and PREREGISTRATION.json are locked before Phase 6. No post-hoc adjustments.
- **Production deployment evaluation:** This is a benchmark experiment on synthetic cases with planted issues. Results do not directly measure production ml-lab performance.
- **Cross-vendor comparison (Phase 10):** Requires operator-configured external API key. Not included in default run.
- **Case authorship or revision:** CASE_VERIFIER decisions go back to the external LLM for regeneration. Claude Code does not author or revise cases.
- **README rewrite optimization:** This README is for the internal experiment record, not for outside-reader presentation.
